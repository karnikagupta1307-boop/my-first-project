ABSTRACT

The Smart Parcel Management System is a full-stack web application designed to digitize and streamline the entire parcel management process within educational institutions. Currently, most institutions rely on paper-based or spreadsheet-driven workflows that are error-prone, time-consuming, and lack any real-time visibility for students.

This system introduces role-based access control for three distinct user types : Admin, Student, and Delivery Person and leverages email-based OTP (One-Time Password) verification to ensure secure and accountable parcel handover. The backend is powered by Node.js and Express, interfacing with a MySQL database to maintain accurate records of students, parcels, and OTP transactions.

Key capabilities include automated email notifications via Nodemailer and Gmail SMTP, real-time status tracking (Arrived ? OTP Generated ? Delivered), a searchable parcel records interface, and a dedicated delivery portal that requires no login. The system eliminates communication gaps, reduces administrative overhead, and establishes a clear digital audit trail for every parcel transaction significantly improving the safety and efficiency of parcel management on campus.













ENTITY - RELATIONSHIP MODEL
ENTITY SET:
1. Students
Description: Represents students registered in the system who are the recipients of parcels.
Attributes:
* id (Primary Key) : Unique auto-incremented identifier for each student. 
* name : Full name of the student. 
* sap_id (Unique) : 11-digit SAP identification number assigned by the institution. 
* phone (Unique) : Registered 10-digit mobile number. 
* email - Email address used for OTP delivery. 
* password_hash : bcrypt-hashed password for student portal login. 
* created_at : Timestamp of account registration. 

2. Parcels
Description: Represents each parcel received at the institution for a registered student.
Attributes:
* id (Primary Key) : Auto-incremented unique parcel record ID. 
* parcel_id (Unique) : System-generated unique parcel identifier. 
* student_id (Foreign Key) : References the student who owns this parcel. 
* student_name : Name of the student (denormalized for quicker display). 
* student_email : Email of the student (denormalized for OTP delivery). 
* source : Origin of the parcel (e.g., Amazon, Flipkart, DTDC). 
* phone : Phone number of the student associated with this parcel. 
* status : Current parcel status: Arrived, OTP Generated, or Delivered. 
* created_at : Timestamp when the parcel was logged into the system. 
* delivered_at : Timestamp when the parcel was marked as delivered. 

3. Parcel OTPs
Description: Represents OTP tokens generated for secure parcel handover verification.
Attributes:
* id (Primary Key) : Auto-incremented unique OTP record ID. 
* parcel_id : The parcel ID for which this OTP was generated. 
* email - The student email to which the OTP was sent. 
* otp_code - The 6-digit randomly generated one-time password. 
* status - OTP lifecycle state: Active, Used, or Expired. 
* created_at - Timestamp when the OTP was generated. 
* expires_at - Timestamp after which the OTP is no longer valid (5-minute window). 
* used_at - Timestamp when the OTP was consumed during verification. 

4. Admins
Description: Represents administrators who manage parcels, verify OTPs, and maintain the system.
Attributes:
* id (Primary Key) - Auto-incremented unique admin ID. 
* username (Unique) - Admin login username. 
* password_hash - bcrypt-hashed password for admin authentication. 

RELATIONSHIP SET:
Student-Parcel Relationship
Description: Each parcel is linked to one registered student. A student may have multiple parcels.
* Cardinality: One-to-Many (One student ? Many parcels) 
* Equation: Student (1) ? Parcel (0..*) 

Parcel-OTP Relationship
Description: Each parcel may have one or more OTP records generated during different handover attempts.
* Cardinality: One-to-Many (One parcel ? Many OTP records) 
* Equation: Parcel (1) ? ParcelOTP (0..*) 

Parcel-Admin Relationship
Description: Admin manages parcel status updates and OTP verifications. An admin can update many parcels.
* Cardinality: Many-to-One (Many parcels ? One admin) 
* Equation: Parcel (0..*) ? Admin (1)

Database & Tables Creation
-- Create and select the database
CREATE DATABASE IF NOT EXISTS parcel_management;
USE parcel_management;

-- Students Table
CREATE TABLE students (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    sap_id VARCHAR(20) UNIQUE NOT NULL,
    phone VARCHAR(15) UNIQUE NOT NULL,
    email VARCHAR(100) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Parcels Table
CREATE TABLE parcels (
    id INT AUTO_INCREMENT PRIMARY KEY,
    parcel_id VARCHAR(20) UNIQUE NOT NULL,
    student_id INT NOT NULL,
    student_name VARCHAR(100) NOT NULL,
    student_email VARCHAR(100) NOT NULL,
    source VARCHAR(100) NOT NULL,
    phone VARCHAR(15) NOT NULL,
    status ENUM('Arrived', 'OTP Generated', 'Delivered') DEFAULT 'Arrived',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    delivered_at TIMESTAMP NULL,
    FOREIGN KEY (student_id) REFERENCES students(id) );

-- Parcel OTPs Table
CREATE TABLE parcel_otps (
    id INT AUTO_INCREMENT PRIMARY KEY,
    parcel_id VARCHAR(20) NOT NULL,
    email VARCHAR(100) NOT NULL,
    otp_code VARCHAR(6) NOT NULL,
    status ENUM('Active', 'Used', 'Expired') DEFAULT 'Active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP NOT NULL,
    used_at TIMESTAMP NULL
);

-- Admins Table
CREATE TABLE admins (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL
);

NORMALIZATIONS 
Normalization is the process of organizing a relational database to reduce redundancy and improve data integrity. It decomposes larger tables into smaller, well-structured ones and establishes clear relationships between them. Below is the normalization analysis applied to the Smart Parcel Management System database.
1NF - First Normal Form
Definition: Every column must contain atomic (indivisible) values, and there must be no repeating groups within a row.
Application:
* All tables in the parcel management database satisfy 1NF. 
* Each column holds a single, atomic value for instance, status in the parcels table stores exactly one value: "Arrived", "OTP Generated", or "Delivered". 
* No arrays or multi-valued attributes are present in any table. 

2NF - Second Normal Form
Definition: The table must satisfy 1NF, and every non-key attribute must be fully dependent on the entire primary key (applicable to composite keys).
Application:
* All tables use a single-column integer primary key (id), so there are no partial dependencies. 
* In the parcels table, all attributes student_name, source, status, created_at are fully dependent on the primary key id. 
* The database satisfies 2NF throughout. 

3NF - Third Normal Form
Definition: The table must satisfy 2NF, and no non-key column should be transitively dependent on the primary key through another non-key column.
Application:
* The parcels table includes student_name and student_email as denormalized fields for performance. While these technically create a transitive dependency via student_id, this is an intentional, documented design trade-off to reduce JOIN overhead in frequent queries. 
* All other attributes in each table maintain direct functional dependency on the primary key with no transitive paths. 
* The database largely adheres to 3NF, with documented exceptions. 
BCNF - Boyce-Codd Normal Form
Definition: A stricter form of 3NF where every determinant in the table must be a candidate key.
Application:
* The students table satisfies BCNF - sap_id and phone are candidate keys with no non-trivial functional dependencies outside candidate keys. 
* The parcel_otps table's otp_code is functionally unique per parcel OTP session, ensuring no BCNF violations. 
* The admins table satisfies BCNF with username serving as a candidate key alongside id. 
Overall, the normalization process applied to the Smart Parcel Management System has produced a well-structured, low-redundancy schema. Intentional denormalizations in the parcels table are performance-driven and documented. Future refactoring may extract these fields into a strict 3NF/BCNF-compliant design as the system scales.
QUERIES 
Query 1 : Using WHERE Clause
Operation: Retrieve all parcels with status 'Arrived'.
SELECT * FROM parcels WHERE status = 'Arrived';

Query 2 : Using WHERE with IN Operator
Operation: Retrieve parcels with status 'OTP Generated' or 'Delivered'.
SELECT * FROM parcels WHERE status IN ('OTP Generated', 'Delivered');

Query 3 : Using WHERE with BETWEEN Operator
Operation: Retrieve parcels received between two specific dates.
SELECT * FROM parcels
WHERE created_at BETWEEN '2024-04-01' AND '2024-05-01';

Query 4 : Using WHERE with NOT IN Operator
Operation: Retrieve all parcels that have not yet been delivered.
SELECT * FROM parcels WHERE status NOT IN ('Delivered');

Query 5 : Using LIMIT Clause
Operation: Retrieve the 5 most recently logged parcels.
SELECT * FROM parcels ORDER BY created_at DESC LIMIT 5;

Query 6 : Using ORDER BY Clause
Operation: Retrieve all parcels ordered alphabetically by student name.
SELECT * FROM parcels ORDER BY student_name ASC;

Query 7 : Using GROUP BY with COUNT
Operation: Count the number of parcels per status category.
SELECT status, COUNT(*) AS total_parcels
FROM parcels
GROUP BY status;

Query 8 : Using HAVING Clause
Operation: Retrieve parcel sources that have more than 2 parcels associated.
SELECT source, COUNT(*) AS total_parcels
FROM parcels
GROUP BY source
HAVING COUNT(*) > 2;

Query 9 : 
Operation: Retrieve parcel details along with the corresponding student's phone and email.
SELECT p.*, s.sap_id, s.phone, s.email
FROM parcels p
INNER JOIN students s ON p.student_id = s.id;

Query 10 : Using LEFT JOIN
Operation: Retrieve all parcels along with their OTP records, even if no OTP has been generated.
SELECT p.*, o.otp_code, o.status AS otp_status, o.expires_at
FROM parcels p
LEFT JOIN parcel_otps o ON p.parcel_id = o.parcel_id;

Query 11 : Using RIGHT JOIN
Operation: Retrieve all OTP records along with their associated parcel details.
SELECT p.parcel_id, p.student_name, o.*
FROM parcels p
RIGHT JOIN parcel_otps o ON p.parcel_id = o.parcel_id;

Query 12 : Using CROSS JOIN
Operation: Retrieve all possible student-admin combinations for audit simulation.
SELECT s.name AS student_name, a.username AS admin_username
FROM students s
CROSS JOIN admins a;

Query 13 : Using EXISTS Subquery
Operation: Retrieve students who have at least one parcel registered.
SELECT * FROM students s
WHERE EXISTS (
    SELECT 1 FROM parcels WHERE student_id = s.id
);

Query 14 : Using NOT EXISTS Subquery
Operation: Retrieve students who have never had any parcel registered.
SELECT * FROM students s
WHERE NOT EXISTS (
    SELECT 1 FROM parcels WHERE student_id = s.id
);

Query 15 : Using Scalar Subquery
Operation: Retrieve total parcel count for each student alongside their details.
SELECT name, sap_id,
    (SELECT COUNT(*) FROM parcels WHERE student_id = s.id) AS total_parcels
FROM students s;


Query 16 : Using Correlated Subquery
Operation: Retrieve parcels that were delivered after the most recently delivered parcel for the same student.
SELECT * FROM parcels p1
WHERE p1.delivered_at > (
    SELECT MAX(p2.delivered_at)
    FROM parcels p2
    WHERE p2.student_id = p1.student_id
    AND p2.id <> p1.id
);

Query 17 : Using Aggregate Function with GROUP BY
Operation: Count total parcels received per month.
SELECT MONTH(created_at) AS month,
       YEAR(created_at) AS year,
       COUNT(*) AS parcels_received
FROM parcels
GROUP BY YEAR(created_at), MONTH(created_at);

Query 18 : Using UNION
Operation: Retrieve a combined list of all student names and all admin usernames.
SELECT name AS entity_name, 'Student' AS role FROM students
UNION
SELECT username AS entity_name, 'Admin' AS role FROM admins;
Query 19 : Using CASE Statement
Operation: Classify parcels as 'Pending Pickup', 'In Progress', or 'Completed' based on their current status.
SELECT parcel_id, student_name, status,
    CASE
        WHEN status = 'Arrived' THEN 'Pending Pickup'
        WHEN status = 'OTP Generated' THEN 'In Progress'
        WHEN status = 'Delivered' THEN 'Completed'
        ELSE 'Unknown'
    END AS status_label
FROM parcels;

Query 20 : Retrieve Active OTPs Nearing Expiry
Operation: Retrieve all OTP records that are still active but will expire within the next 2 minutes.
SELECT * FROM parcel_otps
WHERE status = 'Active'
AND expires_at <= DATE_ADD(NOW(), INTERVAL 2 MINUTE);

Query 21 : Retrieve Parcels from a Specific Source
Operation: Retrieve all parcels that arrived from Amazon.
SELECT * FROM parcels WHERE source = 'Amazon';

Query 22 : Retrieve Confirmed Deliveries for Today
Operation: Retrieve all parcels that were marked as delivered today.
SELECT * FROM parcels
WHERE status = 'Delivered'
AND DATE(delivered_at) = CURDATE();

Query 23 : Group Parcels by Student and Status
Operation: Retrieve parcel count grouped by student and their status.
SELECT student_name, status, COUNT(*) AS total
FROM parcels
GROUP BY student_name, status;

Query 24 : Retrieve Parcels for a Specific Student
Operation: Retrieve all parcels belonging to a specific student by SAP ID.
SELECT * FROM parcels
WHERE student_id = (
    SELECT id FROM students WHERE sap_id = '40010220097'
);

Query 25 : Retrieve Used OTPs with Parcel Details
Operation: Retrieve all OTP records that have been used along with their associated parcel information.
SELECT o.*, p.student_name, p.source
FROM parcel_otps o
INNER JOIN parcels p ON o.parcel_id = p.parcel_id
WHERE o.status = 'Used';

Query 26 : Count Parcels Per Student
Operation: Retrieve each student's name along with their total number of parcels.
SELECT s.name, s.sap_id, COUNT(p.id) AS parcel_count
FROM students s
LEFT JOIN parcels p ON s.id = p.student_id
GROUP BY s.id;

Query 27 : Retrieve Students with a Specific Phone Pattern
Operation: Retrieve students whose phone numbers begin with '987'.
SELECT * FROM students WHERE phone LIKE '987%';

Query 28 : Retrieve Parcel Status Updated with Admin Context
Operation: Retrieve all delivered parcels along with timestamps, ordered by most recent delivery.
SELECT parcel_id, student_name, source, delivered_at
FROM parcels
WHERE status = 'Delivered'
ORDER BY delivered_at DESC;

Query 29 : Retrieve Parcels with Descriptive Status and OTP Info
Operation: Retrieve all parcels with a human-readable status label and whether an OTP has been generated.
SELECT p.parcel_id, p.student_name,
    CASE p.status
        WHEN 'Arrived' THEN 'Parcel Waiting at Counter'
        WHEN 'OTP Generated' THEN 'OTP Sent - Awaiting Verification'
        WHEN 'Delivered' THEN 'Successfully Handed Over'
    END AS status_description,
    IFNULL(o.otp_code, 'N/A') AS otp_info
FROM parcels p
LEFT JOIN parcel_otps o ON p.parcel_id = o.parcel_id
AND o.status = 'Active';

Query 30 : Full Parcel Detail View with Student Contact
Operation: Retrieve complete parcel details including student contact information and OTP history.
SELECT p.parcel_id, p.source, p.status, p.created_at,
       s.name, s.sap_id, s.phone, s.email,
       o.otp_code, o.status AS otp_status, o.expires_at
FROM parcels p
INNER JOIN students s ON p.student_id = s.id
LEFT JOIN parcel_otps o ON p.parcel_id = o.parcel_id;

Result
The Smart Parcel Management System significantly improves parcel handling by providing an efficient, secure, and scalable solution.
* Efficient Tracking: Parcels move through clear stages - Arrived, OTP Generated, and Delivered with real-time visibility for admins. 
* Secure Handover: A time-limited OTP system ensures only the intended student can collect parcels, improving security and accountability. 
* Automated Notifications: OTP emails are sent instantly using Nodemailer, reducing manual effort and delays. 
* Role-Based Access: Separate roles (Admin, Student, Delivery Person) ensure controlled access and organized workflow. 
* Reduced Administrative Work: Auto-fetching student details speeds up parcel entry and minimizes human errors. 
* Scalable Architecture: A well-designed database supports large data volumes while maintaining consistency through relationships and constraints.

Conclusion
The Smart Parcel Management System modernizes traditional paper-based parcel handling in educational institutions. It integrates a MySQL database, Node.js backend, and a user-friendly interface to solve issues like manual tracking, insecure handovers, delayed notifications, and lack of records.
The system uses role-based access for Admin, Student, and Delivery Person, ensuring each user interacts with relevant features. OTP verification via email adds security, while real-time updates keep students informed.
The database follows normalization up to BCNF with some performance-based adjustments. Various SQL queries demonstrate its efficiency and flexibility.
Overall, the project shows how web technologies and database design can effectively digitize and improve campus parcel management.
